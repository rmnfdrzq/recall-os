# 📊 Data Model Specification

RecallOS divides its data modeling into a local-first client-side layout for document metadata and vector search indices, and a server-side relational database for chat sessions and conversational logs.

---

## 📱 Client-Side Data Storage (LanceDB & IndexedDB)

The desktop Tauri client manages local RAG files inside LanceDB under the application data directory:
```text
<app_data_dir>/recallos_lancedb/
```

It maintains two primary tables: `documents` and `document_chunks`.

### 1. `documents` Table
Stores high-level file details, status markers, and AI-generated metadata.

| Field Name | Data Type | Description |
| :--- | :--- | :--- |
| `id` | `VARCHAR` (UUID) | Unique identifier generated by the client. |
| `filename` | `VARCHAR` | The name of the file on disk. |
| `file_type` | `VARCHAR` | File extension (e.g., `pdf`, `md`, `py`). |
| `status` | `VARCHAR` | Current processing state (`processing`, `summarizing`, `indexed_text`, `indexing_vectors`, `processed`, `failed`). |
| `summary` | `TEXT` | AI-generated summary of the document contents. |
| `suggested_title` | `VARCHAR` | AI-generated title based on summary. |
| `category` | `VARCHAR` | AI-generated category tag (e.g. `Development`, `HR`, `Notes`). |
| `tags` | `ARRAY<VARCHAR>` | Customizable user or system tags. |
| `file_path` | `VARCHAR` | Local absolute path of the document on the user's host filesystem. |
| `created_at` | `TIMESTAMP` | Record creation timestamp. |
| `updated_at` | `TIMESTAMP` | Last updated timestamp. |

### 2. `document_chunks` Table
Stores raw text snippets, corresponding dense vector embeddings, and metadata coordinates.

| Field Name | Data Type | Description |
| :--- | :--- | :--- |
| `vector` | `ARRAY<FLOAT32>` (1024) | Dense vector embedding generated by BGE-M3 (1024 dimensions). |
| `text` | `VARCHAR` | The plain text content of this chunk. |
| `document_id` | `VARCHAR` | Foreign relation mapping back to the `documents` table. |
| `metadata` | `JSON` | Nested layout coordinates and NLP tagging. |

#### Chunks `metadata` Payload Structure
The `metadata` field is a flexible JSON object storing structural and linguistic context:
```json
{
  "chunk_index": 0,
  "prev_chunk_index": null,
  "next_chunk_index": 1,
  "page_number": 1,
  "section_title": "1. Executive Summary",
  "section_index": 0,
  "content_type": "paragraph",
  "keywords": ["RAG", "local-first", "Tauri"],
  "entities": {
    "organizations": ["Google DeepMind"],
    "dates": ["May 2026"],
    "technologies": ["Tauri", "React", "Rust"]
  },
  "filename": "recallos_spec.pdf",
  "created_at": "2026-05-26T15:40:00Z"
}
```

### 3. Client-Side Cache (IndexedDB)
Used to cache text chunk embeddings inside the WebView's IndexedDB. The store key is a sha256 hash of the normalized chunk text:
*   **Key**: `sha256(chunk_text)`
*   **Value**: `float32[1024]` embedding array

## ⚙️ Server-Side Data Storage (PostgreSQL via Django)

The `recall-server` backend persists chat histories in PostgreSQL. It does not own or store files or vector tables.

### 1. `ChatSession` Model
Represents a distinct conversation context.

```python
class ChatSession(models.Model):
    id = models.UUIDField(primary_key=True, default=uuid.uuid4, editable=False)
    title = models.CharField(max_length=255, default='New Chat Session')
    created_at = models.DateTimeField(auto_now_add=True)
    updated_at = models.DateTimeField(auto_now=True)
```

### 2. `ChatMessage` Model
Represents an individual message exchanged within a chat session.

```python
class ChatMessage(models.Model):
    ROLE_CHOICES = [
        ('user', 'User'),
        ('assistant', 'Assistant'),
    ]

    session = models.ForeignKey(ChatSession, on_delete=models.CASCADE, related_name='messages')
    role = models.CharField(max_length=10, choices=ROLE_CHOICES)
    content = models.TextField()
    sources = models.JSONField(default=list, blank=True)
    created_at = models.DateTimeField(auto_now_add=True)
```

#### `sources` Field Structure
When the model returns an answer, the server resolves cited reference tags (e.g. `[S1]`) and links them back to their local coordinate maps. The `sources` JSON list is committed to the database in this format:
```json
[
  {
    "document_id": "doc-uuid-1234",
    "filename": "notes.md",
    "suggested_title": "Project Notes",
    "chunk_index": 4,
    "page_number": 1,
    "section_title": "Local Databases",
    "snippet": "RecallOS stores chunks inside a local serverless LanceDB..."
  }
]
```
If the LLM omits reference markers but supplies an answer, the backend populates `sources` with the top unique document contexts received in the request, capped at 4 items.
